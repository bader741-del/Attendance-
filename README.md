## الميزات (النسخة 2.0)

- **توقيع النائب الإداري**: يُحفظ الاسم والتوقيع المرسوم من الإعدادات، ويُختم تلقائياً على كل تقرير معتمد ويظهر في النسخ المطبوعة
- **طباعة / PDF**: للتقارير اليومية والأسبوعية والشهرية مع خانة التوقيع في نهاية التقرير
- **نسخ احتياطي واستعادة**: تنزيل واستعادة جميع البيانات كملف JSON من الإعدادات
- **تنبيهات ذكية**: جرس التنبيهات يعرض التقارير المعلقة، جداول الدوام غير المغطاة اليوم، والنقص الحرج في التغطية
- **سجل تدقيق**: يوثّق كل إجراء (دخول، اعتماد، رفض، إضافة، حذف...) مع إمكانية التصدير
- **كلمة مرور مشفرة**: تُحفظ بتجزئة SHA-256 مع ترحيل تلقائي عند أول دخول
- **وضع ليلي**: زر تبديل في جميع الصفحات
- **PWA**: تثبيت المنصة كتطبيق على الجوال وتعمل دون اتصال (عند نشرها على HTTPS)

## التشغيل المحلي

افتح `index.html` مباشرةً في أي متصفح حديث. لا يحتاج الملف إلى خادم.

## الرفع على GitHub Pages

1. أنشئ مستودعاً جديداً على [github.com](https://github.com)
2. ارفع جميع الملفات إلى المستودع
3. افتح **Settings → Pages** واختر الفرع `main` والمجلد `/root`
4. انقر **Save** — سيظهر الرابط خلال دقيقة تقريباً

```
https://<username>.github.io/<repo-name>/
```

## البيانات والمزامنة السحابية (v2)

النظام **محلي-أولاً**: localStorage هو ذاكرة العمل، وSupabase (إن فُعّل) هو مصدر الحقيقة — سحب تلقائي كل دقيقة ودفع عبر صف انتظار يعيد المحاولة عند انقطاع الاتصال. ما دام `config.js` بقيمه النائبة يعمل كل شيء محلياً كما في v1.

### خطوات تفعيل السحابة

1. أنشئ مشروعاً على [supabase.com](https://supabase.com)
2. SQL Editor ← الصق `supabase-schema.sql` كاملاً ← Run، ثم `supabase-approvals-migration.sql` ← Run، ثم `supabase-rounds-migration.sql` ← Run
3. Authentication ← Add user ← بريد `admin@mksh.local` وكلمة مرور قوية
4. نفّذ سطر تسجيل المسؤول الموجود آخر ملف SQL (بعد تعديل البريد إن غيّرته)
5. Settings ← API ← انسخ Project URL و anon key إلى `config.js` وارفعه
6. ادخل كلوحة مسؤول — ستظهر رسالة رفع البيانات المحلية للسحابة لمرة واحدة

### الأمان

- RLS مفعّل على كل الجداول بسياسات أدوار (`is_admin()`) — لا سياسات مفتوحة
- المجهول يقرأ أسماء الأقسام فقط؛ التحقق من كود الموظف وإدخال التقارير عبر دوال RPC لا تكشف الجداول
- مصادقة المسؤول عبر Supabase Auth، وسجل التدقيق السحابي غير قابل للتعديل أو الحذف

## نظام اعتماد الجولات

### الجولة متعددة الأقسام (بوابة الموظف)

كل العملية داخل صفحة الموظف نفسها — دون أي صفحة أو شاشة أخرى. مدخل البيانات يبدأ جولة واحدة (تاريخ/يوم/فترة/مستشفى) ويضيف داخلها عدداً غير محدود من الأقسام بزر «إضافة قسم آخر» — لكل قسم بياناته (إجمالي/حضور/غياب/منسحبون/مجازون/متغيبون/ملاحظات). قسم **«اعتماد الجولة»** أسفل الصفحة **لا يظهر إلا بعد اكتمال إدخال جميع الأقسام**، ويحوي: اسم المدير المناوب (قائمة منسدلة من قاعدة البيانات) + كوده + التوقيع الإلكتروني + زر «اعتماد الجولة كاملة». الحفظ على خطوتين: زر **«حفظ الجولة» معطل طوال فترة الإدخال**. عند «اعتماد الجولة كاملة» يتحقق الخادم من الكود ومطابقته للاسم المختار، فيتحول قسم الاعتماد إلى الأخضر مع «✓ تم اعتماد الجولة»، وتظهر أعلى الصفحة رسالة «تم اعتماد الجولة بواسطة المدير المناوب ويمكن الآن حفظ الجولة.»، وتُقفل جميع الأقسام (يفكها فقط زر «إلغاء الاعتماد» قبل الحفظ النهائي)، وعندها فقط يتفعّل زر الحفظ. الحفظ النهائي يخزّن الجولة كاملة في Supabase باعتماد واحد على مستوى الجولة (`approval_status = approved`, `approved_by`, `approved_at`, `approval_signature`, `approval_code_used`) — وبدون الاعتماد لا حفظ إطلاقاً (رسالة: «يجب اعتماد الجولة كاملة من المدير المناوب قبل الحفظ»). الهيكل في Supabase: جدول `rounds` للجولة + جدول `round_sections` لأقسامها (`round_id`)، مع نسخة مرآتية معتمدة لكل قسم في `reports` كي تستمر كل التقارير والإحصائيات (التي تعتمد المعتمد فقط افتراضياً) بلا كسر — ملف `supabase-rounds-migration.sql`.

### صفحة اعتماد الجولات (المراجعة اللاحقة)

صفحة `approvals.html` مستقلة لمدير المناوبة والمدير العام فقط (لا تظهر للموظفين):

- **الدخول**: مدير المناوبة بكوده الخاص (يُدار من لوحة المسؤول ← «مدراء المناوبة»)، والمدير العام بجلسة Supabase Auth من لوحة المسؤول
- **الجدول**: رقم الجولة، التاريخ، اليوم، الفترة، المستشفى، القسم، اسم الموظف، وقت الإدخال، والحالة بألوانها (🟡 بانتظار الاعتماد · 🟢 معتمدة · 🔴 مرفوضة) مع عدادات أعلى الصفحة
- **التفاصيل**: الضغط على أي جولة يعرض كامل بيانات الموظف دون تعديل، مع أزرار: اعتماد (توقيع إلكتروني إلزامي يُرسم عند كل اعتماد ويُسجَّل مع الاسم والتاريخ والوقت) · رفض (سبب إلزامي ويعود السجل للموظف) · رجوع للموظف لإعادة التعديل
- بعد الاعتماد لا تُعدَّل الجولة إلا من المدير العام (إلغاء الاعتماد)
- **بوابة الموظف**: تعرض الجولات المرفوضة/المُعادة مع السبب، وبعد تعديلها تعود تلقائياً إلى «بانتظار الاعتماد»
- التقارير اليومية/الأسبوعية/الشهرية تعرض **المعتمدة فقط افتراضياً** مع فلتر لجميع الحالات
- قاعدة البيانات: حقول `approval_status` و`approved_by/at/date/time` و`approval_signature` و`rejected_reason` و`returned_for_edit` (ملف `supabase-approvals-migration.sql`)؛ صلاحيات الموظف ومدير المناوبة تُفرض عبر دوال RPC آمنة (security definer) والمدير العام عبر سياسات RLS

## المرحلة 5 — خريطة عمليات المستشفيات

صفحة `hospital-map.html` (خلف بوابة دخول المسؤول، ومربوطة من قائمة لوحة المسؤول):

- بطاقة تفاعلية لكل مستشفى: نسبة الحضور، عدد الموظفين، حاضر/غائب/إجازة/منسحب، التقارير المعلقة، وقت آخر تقرير
- تلوين الحالة آلياً: 🟢 ممتاز (95%+) · 🟡 يحتاج متابعة (85–94%) · 🔴 حرج (أقل من 85%)
- خريطة Leaflet حقيقية بمواقع المستشفيات الثلاثة (الإحداثيات قابلة للتعديل من `HospitalMap.META` أعلى `hospital-map.js`)
- الضغط على البطاقة أو العلامة يفتح لوحة تفصيلية: إحصائيات الأقسام، رسوم Chart.js (اتجاه الحضور + الحضور حسب القسم)، ترتيب الأقسام (الأفضل أداءً والأكثر غياباً)، وتبويبات يومي/أسبوعي/شهري
- فلاتر زمنية: اليوم / أمس / هذا الأسبوع / هذا الشهر / نطاق مخصص
- تحديث تلقائي عند وصول تقرير جديد: قناة Supabase Realtime في الوضع السحابي، وحدث `storage` محلياً، مع سحب دوري احتياطي كل دقيقة
- كل الأرقام محسوبة آلياً من التقارير الفعلية — لا بيانات ثابتة

## المرحلة 10 — مركز القيادة التنفيذي

صفحة `executive.html` للإدارة العليا (خلف بوابة دخول المسؤول، ومربوطة من قائمة لوحة المسؤول):

- ملخص تنفيذي أعلى اللوحة (لقطة اليوم دائماً، مستقلة عن الفلتر): الحالة العامة، حضور اليوم، التزام اليوم، مستشفيات تحتاج انتباهاً، تقارير معلقة، تنبيهات حرجة
- لوحة «أقسام تتطلب متابعة»: حضور أقل من 85% أو غياب فوق العتبة — الأسوأ أولاً
- 9 مؤشرات تنفيذية: إجمالي الموظفين، الحاضرون، الغائبون، الإجازات، المنسحبون، نسبة الحضور، نسبة الالتزام، التقارير المرفوعة، والمعلقة — مع مؤشر التغيّر عن الفترة السابقة
- مقارنة المستشفيات الثلاثة: حضور % والتزام % وتقارير ناقصة وعدد موظفين وحالة لونية
- 7 لوحات رسوم Chart.js: اتجاه الحضور، مقارنة أسبوعية، مقارنة شهرية (6 أشهر)، ترتيب الأقسام، ترتيب المستشفيات، خريطة حرارية (مستشفيات × أيام)، وعدادات تقدم المؤشرات
- تنبيهات آلية: حضور أقل من 85%، مستشفى بلا تقرير اليوم، ارتفاع الغياب عن الفترة السابقة، أقسام تجاوزت عتبة الغياب (15%)
- الأفضل أداءً: أفضل مستشفى وقسم وأعلى نسبة والأكثر تحسناً
- رؤى إدارية مولّدة آلياً من الأرقام الفعلية (تغيّر الحضور، الأعلى التزاماً، أقسام تتطلب متابعة…)
- فلاتر سريعة: اليوم / أمس / الأسبوع / الشهر / الربع / السنة / نطاق مخصص
- تصدير: Excel متعدد الأوراق، وطباعة / PDF بأنماط طباعة مخصصة
- تحديث لحظي عبر الطبقة المشتركة `analytics-core.js` (Realtime + storage + سحب دوري)

نسبة الالتزام تُحسب آلياً: خانات (يوم × جدول دوام) التي وصلها تقرير مقابل المتوقع من جداول الدوام المعرّفة.

## المرحلة 15 — نظام التقارير التنفيذية

صفحة `exec-reports.html` (خلف بوابة دخول المسؤول، ومربوطة من قائمة لوحة المسؤول):

- 5 أنواع تقارير: يومي، أسبوعي، شهري، ربع سنوي، سنوي — إضافة إلى فترة مخصصة
- فلاتر: المستشفى، القسم، الموظف (مُدخل البيانات)، حالة الاعتماد، مع خيارات تضمين الإجازات والانسحابات
- كل تقرير يتضمن آلياً: ملخصاً تنفيذياً (الأداء العام، أفضل مستشفى/قسم، جهات تتطلب متابعة، تنبيهات حرجة)، مؤشرات الفترة، 6 رسوم Chart.js، ترتيب المستشفيات والأقسام، ملخص أداء نصياً، وملاحظات مولّدة آلياً من الأرقام
- معاينة فورية قبل التصدير بمظهر مستند رسمي (شعار، اسم المنصة، عنوان، تاريخ ووقت الإصدار، ترويسة وتذييل)
- تصدير: Excel متعدد الأوراق (ملخص/مستشفيات/أقسام/ملاحظات)، وطباعة / PDF بتذييل يتكرر على كل صفحة (أرقام الصفحات من خيار «الترويسات والتذييلات» في نافذة الطباعة — قيد معروف في المتصفحات)
- كل الحسابات من الطبقة المشتركة `analytics-core.js` (Analytics.overview / observations / chartCfg) — بلا أي تكرار كود مع لوحة القيادة
